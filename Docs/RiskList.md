# 6. Risk List

*Ranked unknowns and likely failure points, each with the cheapest first-validation.* Ordered by a rough
`impact × uncertainty`. "Validate by" items are designed to be checkable **before** committing to the full
arena.

Status marks: ✅ verified in-game · ◐ partly verified (mechanism confirmed, arena-specific test still open) ·
unmarked = not yet exercised. Verified items cite the `tools/FalseGods.Probe` run (game 6000.3.6f1, A\* 5.3.8).

| **R4** ✅ | **`NavmeshPrefab.Apply()` from a mod + validity of a bundle-baked navmesh** | Option 1 nav (prebaked, deterministic, cheap) hinges on this. If it fails we fall back to runtime rescan (Option 2), which is heavier and needs the `NavMeshCleaner` workaround. | **RESOLVED — Option 1 works end-to-end from a mod (probes P5–P5d, 0.13.0–0.18.0, 2026-07-12/13).** Bake our arena once (`NavmeshPrefab.Scan`) → save the serialized `TileMeshes` bytes → apply them with `Deserialize` + `SnapToGraph` + `ReplaceTiles`; measured: applying the saved `arena-nav-PocRoom-cell0.30.bytes` (151 bytes) to a floated arena took its floor from 0 → walkable nodes, tile dims matched (1×1), and `ClearTiles` restored the graph. The path needs **no A\* in `FalseGods.Unity`** — the plugin (which already has the game's A\*) does the apply; the bundle only ships the bytes. Remaining is build-work, not risk: bytes into the bundle as a `TextAsset`, the arena controller, and one bake per distinct environment `cellSize` (the bake is `cellSize`-specific — `Apply` needs matching tile dims). Multiplayer byte-identical parity is tracked separately (R7/R34); shipping one baked artifact satisfies it by construction. **The road there (kept for the record):** **Option 2 (runtime rescan) RULED OUT for our floor (probe P5, 2026-07-12).** A mod-spawned bundle arena floor — mesh on `Geometry(3)`, `isReadable=true`, `collectionMode=Layers`, and a `RecastNavmeshModifier` (WalkableSurface, `AlwaysInclude`) — produced **zero walkable nodes** under every runtime bake: `UpdateGraphs(bounds)` (only edits existing walkability — `MetalGate`'s use, does not rasterize), a manually-driven `ScanAsync`, and the game's own synchronous `AstarPath.Scan()` (`NavMeshManager.BakeNavMesh`). Each time our floor stayed ~6 m from the nearest node with 0 in-bounds walkable nodes, while a `Scan()` *did* re-rasterize the level (node total changed) and an anchor point flipped level areas walkable — so the scan ran, it just never rasterized our runtime-spawned floor. **Re-test (probe 0.13.0, faithful sequence, 2026-07-12):** replicating `BuildNavMeshNode` exactly — `recastGraph.SnapBoundsToScene()` *then* `Scan()` — still gave **0 nodes** on our floor, and eliminated two candidate causes: (a) **bounds** — the floor sample was already inside `forcedBounds` and `SnapBoundsToScene` changed them by nothing (the level graph already spans a 675×440×673 volume), so out-of-bounds was not it; (b) **collection** — the floor is `isReadable=true` with a Position attribute, so it *passes* `ConvertMeshToGatheredMesh` (in Layers mode it is gathered twice — by its `Geometry(3)` layer and by the `AlwaysInclude` modifier). So the 0-nodes result is a **rasterization/walkability** failure, not a collection failure, and every test so far placed the floor in *occupied* space (floating into level geometry, or overlapping the level floor). **Conclusion unchanged: arena nav must ship a prebaked `NavmeshPrefab` (Option 1, the game's own `BuildNavMeshNode` path)** — and it is the determinism-correct choice for host+client parity (R7/R34) regardless. **Root cause found — our floor mesh had inverted winding (probes P5b/P5c, 0.15.0–0.17.0, 2026-07-12).** The `NavmeshPrefab.Scan` + `RecastGraph.ReplaceTiles` *plumbing* works from a mod (P5b inserted +161 walkable nodes and `ClearTiles` removed them) — but P5b's "our floor is walkable" was a **false positive**: that arena sat at ground level, so the nodes it applied were *level* geometry in the arena footprint, not our floor. An isolated bake in genuinely clear space (+300 m, P5c) baked our floor to **0 triangles**, and a side-by-side test pinned the cause: our floor's top-face **geometric normal points down** (`cross(v1-v0,v2-v0) = (0,-1,0)`), because `PocRoomGenerator.BuildBoxMesh` wound the box inside-out. A* recast decides walkability from that geometric winding (not the supplied vertex normals — which is why the floor still *rendered*), so it read the floor top as a ceiling → 0 walkable triangles. A winding-reversed copy of the same mesh baked correctly (normal `(0,1,0)`, 2 triangles). **Fix (verified in-game):** corrected the winding in `BuildBoxMesh` and rebuilt the bundle; the arena now bakes **16 triangles** (top-face normal `(0,1,0)`) and P5c wrote `arena-nav-PocRoom-cell0.30.bytes`. **P5d then closed the loop:** loading that saved artifact and applying it took the floated arena's floor from 0 → walkable — so a baked-once artifact makes our floor walkable from a mod, and it **survived without adding a `NavMeshCleaner` anchor** (the `ReplaceTiles` path side-steps the cleaner, as §4.3 predicted — see R5). Note `Apply()` requires the baked tile dimensions to match the runtime graph, and `cellSize` varies per environment (0.1 Fortress, 0.3 Desert), so the arena bake is `cellSize`-specific (ship one variant per distinct environment `cellSize`). |
| **R5** ✅ | **`NavMeshCleaner` flood-fill discards our island** | On a runtime rescan, walkability is restricted to areas containing `validNavMeshPoints` (connectors + `navMeshAnchor`s). A custom arena with none gets marked unwalkable. `NavMeshCleaner.OnGraphsPostUpdate` marks every node whose `Area` is not reached from a `validNavMeshPoint` as `Walkable = false`. | **Mechanism confirmed (probe P0), arena survival still to test.** On a live level the active cleaner had exactly **3** `validNavMeshPoints`, all from the single `Room`'s **3** `navMeshAnchors`; the scan's 1492 nodes were all walkable, i.e. the cleaner keeps only anchor-reachable areas. An additive arena supplies **zero** anchors unless we add one. **Cleaner reconfirmed on our arena (probe P5, 2026-07-12):** appending a `validNavMeshPoint` and re-baking flipped a whole area walkable (+216 nodes), so the cleaner does keep only anchor-reachable areas as modelled. But **our own floor could not be tested for survival because it never rasterized under any runtime rescan (see R4, reconfirmed with the faithful `SnapBoundsToScene()` + `Scan()` sequence in probe 0.13.0)** — there were no island nodes for the cleaner to keep or erase. Floor survival therefore waited on the prebaked `NavmeshPrefab` (Option 1) path. **Resolved there (probe P5d, 2026-07-13):** once the floor-mesh winding was fixed (R4), applying the baked artifact via `ReplaceTiles` left our floated arena floor **walkable without adding any anchor** — the tile-replace path does not go through the `NavMeshCleaner` flood-fill that a full scan triggers, exactly the side-step §4.3 anticipated. So the cleaner is a concern only for Option 2 (runtime rescan), which we are not using. |
| **R6** ◐ | **Shader variants missing** for reused/own materials | Pink materials in-game even if fine in editor. | **Split result (probe P1 + visible P3, 2026-07-12).** *Vanilla reuse — good:* the **visible** vanilla prefab `CaveGrubGrub` (430 renderers, **0 null materials**; shaders `Shader Graphs/MasterShader`, `…/CharacterShader`, `…/StaticCharacterShader`, `…/MasterParticleShader`, `ProBuilder6/Standard Vertex Color`, all `supported`) rendered **correctly — no pink** — under our own `LightingRoot`. | **R8** ✅ | **Teardown pollutes the level we are still in** (stale nav nodes / leaked objects / unreleased Addressables) | `AstarPath.active` is shared global state for the **active** level. A normal level change rebuilds it (`ClearGraphs()`+`Destroy` at `GameManager.cs:1097`, `Instantiate(astarPathPrefab)` at `:1137`), so residue does not cross levels — but every vanilla NPC paths on our leftovers until that transition happens, and a Boss Rush or hub-return flow may never reach one. | **VERIFIED (probe P7, 0.24.2, in-game 2026-07-13, Act_03_Desert).** Teardown leaves the active level clean, but only with the right removal step. **Same-level:** the run floats the arena onto a footprint tile the player stands on, snapshots that tile (geometry + per-node `Walkable`), applies our arena nav — which **clobbers the level ground there, 64 → 0** (`RecastGraph.ReplaceTiles`/`ClearTiles` operate on whole XZ tiles across the full Y column) — then restores it. **A `ClearTiles`-only teardown, which is all A\*'s own `NavmeshPrefab.OnDisable` does, would leave that 38.4 m hole.** The clean teardown is `ReplaceTiles(snapshotted level tiles)` **plus reapplying the saved `Walkable` flags**: `ReplaceTiles` rebuilds nodes walkable-by-geometry and does **not** re-run the `NavMeshCleaner` flood-fill (the same side-step P5d relied on at R5), so geometry-only restore *over-restores* (first run: footprint 25 → 131); reapplying the flags returns the footprint to **exactly baseline** (level-ground 64 → 64, arena-band 0 → 0, whole-graph 2704 → 2690 within tolerance), with **0 arena `GameObject`s** and the bundle unloaded. **Cross-level:** the game rebuilds `AstarPath.active` per level (a fresh instance each load — P0), so residue cannot cross a level change; confirmed by F10 on the next level. **Design consequence:** the arena controller's teardown owner must snapshot + restore the overwritten tiles (geometry + walkability), never merely clear them (R30). |
| **R9** ◐ | **Big-boss pathing** doesn't fit a normal recast agent | A large original boss may clip walls / stick on corners with plain `RichAI`. | **Ordinary-enemy A\* pathing on our arena — VERIFIED (probe P6, 0.23.0, 2026-07-13).** With our baked navmesh applied to a floated, isolated copy of the arena (P5c bake in clear space + P5d `ReplaceTiles`), two checks passed on **10 environments** (Act_01 Castle/Sewers/Shanty/Caves, Act_02 Bridge/Forest/Fortress, Act_03 Desert/EndChurch — including the coarse-tile DesertBoss/Forest that first failed): (a) an `ABPath` between the EnemySpawn (7,7) and PlayerSpawn (-7,-7) corners — whose straight line crosses the central pillar — **routes AROUND it** (closest approach ~2.8 m vs the 1.0 m pillar footprint, path complete, 12 waypoints); (b) a **real vanilla `Npc`** (`HellshrewSticka`, resolved + spawned + registered + activated the game's own way — `UnitId.GetAsset()` → `FetchAndLoadUnitLoader()` → `SetStats` → `Spawn()` → `GameManager.npcs` → `ActivateBehaviour`, driven by `Npc.SetForcedDestination` with the behaviour tree off) **walks ~16 m around the pillar** to the far corner, staying ~1 m clear.
