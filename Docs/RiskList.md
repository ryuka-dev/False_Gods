| **R4** ‚úÖ | **`NavmeshPrefab.Apply()` from a mod + validity of a bundle-baked navmesh** | Option 1 nav (prebaked, deterministic, cheap) hinges on this. If it fails we fall back to runtime rescan (Option 2), which is heavier and needs the `NavMeshCleaner` workaround. | **RESOLVED ‚Äî Option 1 works end-to-end from a mod (probes P5‚ÄìP5d, 0.13.0‚Äì0.18.0, 2026-07-12/13).** Bake our arena once (`NavmeshPrefab.Scan`) ‚Üí save the serialized `TileMeshes` bytes ‚Üí apply them with `Deserialize` + `SnapToGraph` + `ReplaceTiles`; measured: applying the saved `arena-nav-PocRoom-cell0.30.bytes` (151 bytes) to a floated arena took its floor from 0 ‚Üí walkable nodes, tile dims matched (1√ó1), and `ClearTiles` restored the graph. The path needs **no A\* in `FalseGods.Unity`** ‚Äî the plugin (which already has the game's A\*) does the apply; the bundle only ships the bytes. Remaining is build-work, not risk: bytes into the bundle as a `TextAsset`, the arena controller, and one bake per distinct environment `cellSize` (the bake is `cellSize`-specific ‚Äî `Apply` needs matching tile dims). Multiplayer byte-identical parity is tracked separately (R7/R34); shipping one baked artifact satisfies it by construction. **The road there (kept for the record):** **Option 2 (runtime rescan) RULED OUT for our floor (probe P5, 2026-07-12).** A mod-spawned bundle arena floor ‚Äî mesh on `Geometry(3)`, `isReadable=true`, `collectionMode=Layers`, and a `RecastNavmeshModifier` (WalkableSurface, `AlwaysInclude`) ‚Äî produced **zero walkable nodes** under every runtime bake: `UpdateGraphs(bounds)` (only edits existing walkability ‚Äî `MetalGate`'s use, does not rasterize), a manually-driven `ScanAsync`, and the game's own synchronous `AstarPath.Scan()` (`NavMeshManager.BakeNavMesh`). Each time our floor stayed ~6 m from the nearest node with 0 in-bounds walkable nodes, while a `Scan()` *did* re-rasterize the level (node total changed) and an anchor point flipped level areas walkable ‚Äî so the scan ran, it just never rasterized our runtime-spawned floor. **Re-test (probe 0.13.0, faithful sequence, 2026-07-12):** replicating `BuildNavMeshNode` exactly ‚Äî `recastGraph.SnapBoundsToScene()` *then* `Scan()` ‚Äî still gave **0 nodes** on our floor, and eliminated two candidate causes: (a) **bounds** ‚Äî the floor sample was already inside `forcedBounds` and `SnapBoundsToScene` changed them by nothing (the level graph already spans a 675√ó440√ó673 volume), so out-of-bounds was not it; (b) **collection** ‚Äî the floor is `isReadable=true` with a Position attribute, so it *passes* `ConvertMeshToGatheredMesh` (in Layers mode it is gathered twice ‚Äî by its `Geometry(3)` layer and by the `AlwaysInclude` modifier). So the 0-nodes result is a **rasterization/walkability** failure, not a collection failure, and every test so far placed the floor in *occupied* space (floating into level geometry, or overlapping the level floor). **Conclusion unchanged: arena nav must ship a prebaked `NavmeshPrefab` (Option 1, the game's own `BuildNavMeshNode` path)** ‚Äî and it is the determinism-correct choice for host+client parity (R7/R34) regardless. **Root cause found ‚Äî our floor mesh had inverted winding (probes P5b/P5c, 0.15.0‚Äì0.17.0, 2026-07-12).** The `NavmeshPrefab.Scan` + `RecastGraph.ReplaceTiles` *plumbing* works from a mod (P5b inserted +161 walkable nodes and `ClearTiles` removed them) ‚Äî but P5b's "our floor is walkable" was a **false positive**: that arena sat at ground level, so the nodes it applied were *level* geometry in the arena footprint, not our floor. An isolated bake in genuinely clear space (+300 m, P5c) baked our floor to **0 triangles**, and a side-by-side test pinned the cause: our floor's top-face **geometric normal points down** (`cross(v1-v0,v2-v0) = (0,-1,0)`), because `PocRoomGenerator.BuildBoxMesh` wound the box inside-out. A* recast decides walkability from that geometric winding (not the supplied vertex normals ‚Äî which is why the floor still *rendered*), so it read the floor top as a ceiling ‚Üí 0 walkable triangles. A winding-reversed copy of the same mesh baked correctly (normal `(0,1,0)`, 2 triangles). **Fix (verified in-game):** corrected the winding in `BuildBoxMesh` and rebuilt the bundle; the arena now bakes **16 triangles** (top-face normal `(0,1,0)`) and P5c wrote `arena-nav-PocRoom-cell0.30.bytes`. **P5d then closed the loop:** loading that saved artifact and applying it took the floated arena's floor from 0 ‚Üí walkable ‚Äî so a baked-once artifact makes our floor walkable from a mod, and it **survived without adding a `NavMeshCleaner` anchor** (the `ReplaceTiles` path side-steps the cleaner, as ¬ß4.3 predicted ‚Äî see R5). Note `Apply()` requires the baked tile dimensions to match the runtime graph, and `cellSize` varies per environment (0.1 Fortress, 0.3 Desert), so the arena bake is `cellSize`-specific (ship one variant per distinct environment `cellSize`). |
| **R5** ‚úÖ | **`NavMeshCleaner` flood-fill discards our island** | On a runtime rescan, walkability is restricted to areas containing `validNavMeshPoints` (connectors + `navMeshAnchor`s). A custom arena with none gets marked unwalkable. `NavMeshCleaner.OnGraphsPostUpdate` marks every node whose `Area` is not reached from a `validNavMeshPoint` as `Walkable = false`. | **Mechanism confirmed (probe P0), arena survival still to test.** On a live level the active cleaner had exactly **3** `validNavMeshPoints`, all from the single `Room`'s **3** `navMeshAnchors`; the scan's 1492 nodes were all walkable, i.e. the cleaner keeps only anchor-reachable areas. An additive arena supplies **zero** anchors unless we add one. **Cleaner reconfirmed on our arena (probe P5, 2026-07-12):** appending a `validNavMeshPoint` and re-baking flipped a whole area walkable (+216 nodes), so the cleaner does keep only anchor-reachable areas as modelled. But **our own floor could not be tested for survival because it never rasterized under any runtime rescan (see R4, reconfirmed with the faithful `SnapBoundsToScene()` + `Scan()` sequence in probe 0.13.0)** ‚Äî there were no island nodes for the cleaner to keep or erase. So the cleaner is a concern only for Option 2 (runtime rescan), which we are not using. |
